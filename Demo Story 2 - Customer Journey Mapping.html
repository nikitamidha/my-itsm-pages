The story arc — what ran, in order, and the handoffs

1Report
Elena → Intake Agent
Intake Agent captures Elena's plain‑language complaint and opens the structured incident record. No forms, no menus. Autonomous
↓ hands off a new incident to triage
2Classify & initial diagnosis
Intake Agent → Diagnostic Agent
Intake Agent classifies the incident — Network Connectivity · P1 · single clinic · critical apps — and invokes the Diagnostic Agent once for a remote assessment before deciding the path. Autonomous
⟲Why it can't be fixed from the backend. General browsing works, but the dental‑app traffic is degraded — the fault sits between local CPE, on‑site cabling and the provider path. Remote telemetry can't isolate which side is at fault; the only definitive test is a physical demarc check on site. With network confidence under the ~70% autonomous ceiling, the Diagnostic Agent concludes: remote resolution exhausted → on‑site required.
↓ hands off a classified ticket flagged "on‑site required" to dispatch
3Route & dispatch
Field Service Orchestrator
Field Service Orchestrator packages the field brief — circuit ID, CPE, backup link, demarc, site history — and assigns Ray Oak, Field Service Agent with a 30‑min ETA. Autonomous
👁Sarah Rose, Customer Service Agent.Watch Monitors the autonomous dispatch from her command center to time the client update — no action unless the agent raises Inputs Required.
✓Ray Oak, Field Service Agent.On‑site Reviews the pre‑arrival job brief on FSL mobile and accepts the job before travelling.
↓ Ray accepts → travels to site to investigate
4Investigate on site
Ray Oak, Field Service Agent
Checks the area node first, then visits the clinic and runs guided runbooks — Area Node Check, Site Edge Health, Clinic App Brownout. Finds: hardware healthy, the app traffic is on a degraded route. On‑site
↓ continues to mitigation — same engineer
5Mitigate
Ray Oak, Field Service Agent
Runs the Failover Mitigation runbook — moves only critical workflows to the backup path, pauses guest Wi‑Fi. Clinic is operating again. (Reversible, so no approval needed.) On‑site
↓ hands off field evidence to diagnosis
6Diagnose & propose fix
Diagnostic Agent
Diagnostic Agent interprets the field evidence, identifies the real routing fault, and proposes the Permanent Route Correction — then raises an approval card for IT Ops. Autonomous
↓ hands off a proposed production change for approval
7Approve
Marcus Chen, IT Ops
Reviews the evidence and approves the route‑policy change inside the same incident — required because it affects clinical and payment traffic across clinics. Approval
↓ hands off an approved change to autonomous execution
8Execute permanent fix
Resolution Agent
Resolution Agent runs the approved Permanent Route Correction runbook — applies the route change, tests the dental apps, monitors stability for 30 minutes, rollback armed. Autonomous
↓ hands off a verified fix to communicate
9Verify & restore
Communications Agent → Elena
Communications Agent confirms restoration to Elena — check‑in, X‑rays, payments and phones all stable, inside the 2‑hour SLA — and escalates the provider issue with evidence. Autonomous

The humans in the loop — and their action mode

Everything else runs autonomously. A human enters at exactly three points, each in a defined mode from the HITL framework — to watch, to do physical work, or to govern a production change.
